## Day 205（周二）— Token 鉴权工程化：过期、刷新（了解）、错误码标准化

**学什么**

- token 过期与登出：服务端如何让 token 失效；客户端如何处理 401。
- 错误码契约：前端/调用方该怎么识别“未登录/已过期/无权限”。

**做什么**

1. 明确 token 策略（写进 README）：
   - 过期时间（例如 2h）
   - logout 行为（立即失效）
   - （可选）同账号多端登录是否互踢
2. 统一错误码：
   - `AUTH_REQUIRED`（401）
   - `AUTH_EXPIRED`（401，可选）
   - `FORBIDDEN`（403，可选）
3. 更新 OpenAPI：明确 token 放哪里（header 名称、示例）。

**验收**

- 401 的错误体稳定；README/Swagger 里能直接按示例调通。

## Day 206（周三）— 事件发布可靠性：Outbox（推荐最小版）

**学什么**

- 为什么“DB 写入成功但消息没发出去”是典型问题。
- **Outbox Pattern**：先把事件写入本地表，再由后台任务投递 MQ（最终一致）。

**做什么**

1. 加一张 `outbox_events`（最小字段）：
   - `id, event_type, payload_json, status, created_at, next_retry_at, retry_count`
2. 在“评论/点赞成功”事务里：**写业务表 + 写 outbox**（同一事务）。
3. 写一个定时任务（`@Scheduled`）：
   - 拉取待投递事件 → publish 到 MQ → 标记已投递
   - 失败则更新 `retry_count/next_retry_at`

**验收**

- 人为让 MQ 断开时：业务仍可写入（outbox 里堆积）；MQ 恢复后事件能补投递。

## Day 207（周四）— 消费可靠性：幂等 + 失败处理（DLQ 或失败表）

**学什么**

- 消费端幂等的常见做法：UNIQUE、幂等表、去重 key。
- 失败处理最低配：DLQ 或落失败表，便于重放。

**做什么**

1. 幂等策略收口（选一个并统一）：
   - 通知表 `dedup_key UNIQUE`（继续用，并补齐所有事件）
2. 失败处理（二选一）：
   - A：DLQ（死信队列）+ `replay` 脚本
   - B：`failed_messages` 表 + 管理接口（dev-only）

**验收**

- 同一事件重复投递不重复落库；消费异常能进入可追踪的失败通道。

## Day 208（周五）— Redis 落地：限流 / 幂等 / 分布式锁（三选一）

**要求**：只选一个做“能演示、能解释”的版本。

### 选项 A（推荐）：限流（防刷点赞/评论）

- 对 `POST /like` 或 `POST /comments`：同用户对同 post **每 10 秒最多 3 次**（示例规则）。
- 用 Redis `INCR + EXPIRE` 实现滑动窗口的简化版。

### 选项 B：幂等 Token（防重复提交）

- 对发帖/评论：客户端带 `Idempotency-Key`，服务端用 Redis 记 5 分钟并返回同一结果。

### 选项 C：分布式锁（了解为主）

- 用 `SET NX PX` 做一个最小锁，保护某个敏感更新（例如批量刷新缓存）。

**验收**

- 写入 README：规则是什么、key 怎么设计、TTL 多久、失败返回什么错误码。

## Day 209（周六）— 可观测：链路日志 + 统计指标（最小可用）

**学什么**

- 为什么需要“能看见”：排查重复/丢失/延迟。

**做什么**

1. 为“评论/点赞 → outbox → publish → consume → 通知落库”链路打结构化日志（至少包含 `eventId/dedupKey/postId/userId`）。
2. 增加一个最小统计：
   - 成功消费数、失败消费数（内存计数 + actuator info，或日志统计均可）

**验收**

- 出问题时你能从日志定位：事件有没有进 outbox、有没有 publish、consumer 有没有处理成功。
